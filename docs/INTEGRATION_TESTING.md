# Integration Testing Guide

This document describes how to run integration tests for the `kata-raw-nn` command implementation, which cross-validates the Swift implementation against KataGo's reference output.

## Overview

The integration testing framework compares the Swift `rawNN()` implementation output with reference files generated by the KataGo executable. This ensures that the Swift implementation produces output that exactly matches KataGo's C++ implementation.

## Architecture

```
┌───────────────────┐
│ Reference Script  │ → Generates KataGo output files
└───────────────────┘
         │
         ▼
┌───────────────────┐
│ Swift kata-raw-nn │ → Implements command formatting
│   Implementation  │
└───────────────────┘
         │
         ▼
┌───────────────────┐
│ Integration Tests │ → Compares Swift output with reference files
└───────────────────┘
```

## Prerequisites

1. **KataGo Executable**: Built with Core ML backend support
   - The script will automatically build it if not found
   - Or use `--force-rebuild` to rebuild from source

2. **Binary Models**: 
   - AI model: `kata1-b28c512nbt-adam-s11165M-d5387M.bin.gz` (~258 MB)
   - Human SL model: `b18c384nbt-humanv0.bin.gz` (for 20k model)
   - Automatically downloaded by the script if not present

3. **Core ML Models**: 
   - `KataGoModel19x19fp16-adam-s11165M.mlpackage` (AI model)
   - `KataGoModel19x19fp16m1.mlpackage` (Human SL model)
   - Must be available in `Sources/KataGoOnAppleSilicon/Models/Resources/`
   - Download from [releases](https://github.com/ChinChangYang/KataGo/releases/tag/v1.16.4-coreml1)

4. **Build Tools**: 
   - Ninja: `brew install ninja`
   - Xcode (for building KataGo)

## Step 1: Generate Reference Files

Reference files are generated using the `generate_kata_raw_nn_reference.sh` script, which:

1. Builds KataGo from source (if needed)
2. Downloads the binary model (if needed)
3. Runs KataGo GTP session with `kata-raw-nn` command
4. Extracts and saves the output to reference files

### Generating Reference Files

```bash
# Generate reference file for AI model (empty board, symmetry 0)
./Scripts/generate_kata_raw_nn_reference.sh

# Generate reference file for 20k human SL model
./Scripts/generate_kata_raw_nn_reference.sh --model-type 20k

# Force rebuild KataGo before generating reference
./Scripts/generate_kata_raw_nn_reference.sh --force-rebuild

# Generate 20k reference with force rebuild
./Scripts/generate_kata_raw_nn_reference.sh --model-type 20k --force-rebuild
```

### Reference File Location

Reference files are stored in:
```
Tests/KataGoOnAppleSiliconIntegrationTests/ReferenceOutputs/
```

Generated files:
- AI model: `kata_raw_nn_empty_board_symmetry_0.txt`
- 20k model: `kata_raw_nn_empty_board_symmetry_0_20k.txt`

## Step 2: Run Integration Tests

### Running All Integration Tests

```bash
swift test --filter KataGoOnAppleSiliconIntegrationTests
```

### Running Specific Test

```bash
# Test empty board (AI model)
swift test --filter KataRawNNIntegrationTests.testKataRawNNEmptyBoard

# Test symmetry 0 (AI model)
swift test --filter KataRawNNIntegrationTests.testKataRawNNSymmetry0

# Test empty board with 20k human SL model
swift test --filter KataRawNNIntegrationTests.testKataRawNNEmptyBoard20k
```

## Test Structure

### Test Files

- **`KataRawNNIntegrationTests.swift`**: Main integration test suite
  - `testKataRawNNEmptyBoard()`: Compares empty board output (AI model)
  - `testKataRawNNSymmetry0()`: Verifies symmetry 0 output format (AI model)
  - `testKataRawNNEmptyBoard20k()`: Compares empty board output (20k human SL model)

### Test Workflow

1. **Load Reference File**: Reads the reference file from `ReferenceOutputs/` directory
2. **Generate Swift Output**: Calls `rawNN()` method with the same parameters
3. **Compare Outputs**: Compares line-by-line with floating-point tolerance
4. **Report Results**: Provides detailed mismatch information if differences are found

### Comparison Strategy

The tests use a relative tolerance-based comparison for floating-point values:

- **Exact Match**: For non-numeric lines (headers, labels)
- **Relative Tolerance Match**: For floating-point values using ratio-based comparison
  - Formula: `abs(swift - ref) / max(abs(swift), abs(ref), 1.0) <= tolerance`
  - Default tolerance: 0.001 (0.1% relative difference)
  - This scales with value magnitude (e.g., 0.04 difference in 4578 is ~0.00087%)
- **Structure Verification**: Ensures both outputs have the same structure (policy grid, ownership grid, etc.)

**Why Relative Tolerance?**
- Values span different scales (probabilities ~0.5, scores ~2-3, squared scores ~4500+)
- Absolute tolerance would be too strict for large values or too loose for small values
- Relative tolerance ensures consistent precision across all value ranges

## Understanding Test Results

### Successful Test

```
Test Case 'testKataRawNNEmptyBoard' passed
```

This means:
- Reference file was found and loaded
- Swift output was generated successfully
- Outputs match within tolerance

### Failed Test

If a test fails, you'll see:

```
Test Case 'testKataRawNNEmptyBoard' failed
Mismatches found:
  Line 5: Swift=whiteLead 0.513, Reference=whiteLead 0.512 (diff: 0.001)
```

This indicates:
- A mismatch was found at the specified line
- The difference exceeds the tolerance threshold
- You may need to adjust the Swift implementation or regenerate the reference file

## Troubleshooting

### Issue: Reference File Not Found

**Error**: `IntegrationTestError.referenceFileNotFound`

**Solution**:
1. Run the reference generation script:
   ```bash
   ./Scripts/generate_kata_raw_nn_reference.sh
   ```
2. Verify the file exists:
   ```bash
   ls Tests/KataGoOnAppleSiliconIntegrationTests/ReferenceOutputs/
   ```

### Issue: KataGo Executable Not Found

**Error**: Script fails to find or build KataGo

**Solution**:
1. Ensure build tools are installed:
   ```bash
   brew install ninja
   ```
2. Force rebuild:
   ```bash
   ./Scripts/generate_kata_raw_nn_reference.sh --force-rebuild
   ```

### Issue: Core ML Backend Not Enabled

**Error**: `Dummy neural net backend: NeuralNet::loadModelFile unimplemented`

**Solution**:
1. The executable was built without Core ML support
2. Rebuild with `--force-rebuild`:
   ```bash
   ./Scripts/generate_kata_raw_nn_reference.sh --force-rebuild
   ```

### Issue: Model Files Not Found

**Error**: `Core ML model not found` or `Binary model not found`

**Solution**:
1. **Core ML Model**: Download from [releases](https://github.com/ChinChangYang/KataGo/releases/tag/v1.16.4-coreml1) and place in `Sources/KataGoOnAppleSilicon/Models/Resources/`
2. **Binary Models**: The script will automatically download them, or manually download from:
   - AI model: `https://media.katagotraining.org/uploaded/networks/models/kata1/kata1-b28c512nbt-adam-s11165M-d5387M.bin.gz`
   - Human SL model: `https://media.katagotraining.org/uploaded/networks/models_extra/b18c384nbt-humanv0.bin.gz`

### Issue: Test Output Mismatches

**Possible Causes**:
1. **Formatting Differences**: Check that Swift output matches KataGo's exact format (precision, spacing)
2. **Value Calculation Errors**: Verify that value extraction from model outputs is correct
3. **Coordinate System Issues**: Ensure policy and ownership grids use correct coordinate mapping
4. **Stale Reference File**: Regenerate reference file if KataGo version changed

**Solution**:
1. Compare raw outputs:
   ```bash
   # Generate Swift output manually
   swift test --filter KataRawNNIntegrationTests.testKataRawNNEmptyBoard
   
   # Compare with reference
   diff -u Tests/.../ReferenceOutputs/kata_raw_nn_empty_board_symmetry_0.txt <swift_output>
   ```
2. Check Swift implementation formatting in `KataGoInference.swift` `rawNN()` method
3. Regenerate reference file if needed

## Human SL Model Support

The integration tests support both AI models and human SL models (20k profile). Human SL models require additional `input_meta` encoding:

### Input Meta Encoding

Human SL models require a 192-element `input_meta` array that encodes:
- Player ranks and human status
- Game metadata (rated/unrated, time control, date, source)
- The encoding follows KataGo's `SGFMetadata::fillMetadataRow()` implementation

The Swift implementation includes:
- `SGFMetadata` struct matching KataGo's C++ structure
- `SGFMetadata.getProfile()` to parse profile names (e.g., "preaz_20k")
- `SGFMetadata.fillMetadataRow()` to generate the 192-element array

### 20k Model Test

The `testKataRawNNEmptyBoard20k()` test:
- Loads the 20k human SL model
- Generates `input_meta` with profile "preaz_20k"
- Compares output against reference generated with `--model-type 20k`

## Adding New Test Cases

To add a new test case:

1. **Generate Reference File**:
   ```bash
   # For AI model
   ./Scripts/generate_kata_raw_nn_reference.sh
   
   # For 20k model
   ./Scripts/generate_kata_raw_nn_reference.sh --model-type 20k
   
   # Modify the script to generate reference for your test case
   # Or manually run KataGo GTP and save output
   ```

2. **Add Test Function**:
   ```swift
   @Test func testKataRawNNSingleMove() async throws {
       let referenceOutput = try loadReferenceFile(testCase: "single_move", symmetry: 0)
       
       let katago = KataGoInference()
       try katago.loadModel(for: "AI")
       
       let board = Board()
       _ = board.playMove(at: Point(x: 3, y: 3), stone: .black)
       let boardState = BoardState(board: board)
       
       let swiftOutput = try katago.rawNN(
           board: board,
           boardState: boardState,
           profile: "AI",
           whichSymmetry: 0
       )
       
       let tolerance = 0.001  // 0.1% relative tolerance
       let result = compareOutputs(swift: swiftOutput, reference: referenceOutput, tolerance: tolerance)
       #expect(result.matches)
   }
   ```

3. **Update Reference Generation Script** (if needed):
   - Add new test case to the script
   - Generate reference file with appropriate naming
   - Support `--model-type` if testing human SL models

## Best Practices

1. **Always Regenerate Reference Files** when:
   - KataGo version changes
   - Model files are updated
   - Output format changes

2. **Use `--force-rebuild`** when:
   - KataGo build fails
   - Core ML backend errors occur
   - After updating KataGo source

3. **Verify Reference Files** before committing:
   - Check file size and line count
   - Verify format matches expected structure
   - Ensure all required sections are present

4. **Keep Reference Files in Git** (optional):
   - Reference files are gitignored by default
   - Consider committing them for CI/CD if needed
   - Or document the generation process clearly

## Continuous Integration

For CI/CD pipelines:

1. **Generate Reference Files** as part of setup:
   ```bash
   ./Scripts/generate_kata_raw_nn_reference.sh --force-rebuild
   ```

2. **Run Integration Tests**:
   ```bash
   swift test --filter KataGoOnAppleSiliconIntegrationTests
   ```

3. **Handle Missing Models**: 
   - Download Core ML models in CI setup
   - Cache binary models between runs
   - Skip tests if models unavailable (graceful degradation)

## Summary

Integration testing for `kata-raw-nn` follows this workflow:

1. **Generate Reference**: Use `generate_kata_raw_nn_reference.sh` to create reference files
2. **Run Tests**: Execute integration tests with `swift test`
3. **Compare Outputs**: Tests automatically compare Swift output with reference files
4. **Fix Issues**: Address any mismatches in the Swift implementation
5. **Iterate**: Regenerate references and re-test as needed

This ensures the Swift implementation produces output that exactly matches KataGo's C++ implementation, providing confidence in the correctness of the `kata-raw-nn` command implementation.
